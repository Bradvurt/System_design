## Вариант 13: Система бронирования отелей (аналог Booking.com)

Получить навык описания архитектуры в стиле **Architecture as Code** и проектирования системы «сверху вниз» с использованием методологии C4 и языка Structurizr DSL.

## 1. Контекст системы (C1)

На этом уровне система представлена как «чёрный ящик». Определены следующие **роли пользователей**:

- **Гость** – неавторизованный пользователь, ищет отели по городу, просматривает список отелей.
- **Зарегистрированный пользователь (Клиент)** – создаёт, просматривает и отменяет свои бронирования.
- **Менеджер отеля** – управляет информацией и бронированиями только своего отеля.
- **Администратор сервиса** – обладает полными правами: управление пользователями, отелями, менеджерами, мониторинг.

**Внешние системы**, с которыми взаимодействует наша система:

- **Email-сервис** – отправка уведомлений (подтверждения бронирования, напоминания).
- **Платежная система** – внешний шлюз для обработки платежей (например, Stripe, PayPal).
- **Картографический сервис** – Google Maps (или OpenStreetMap) для отображения карт и геокодирования адресов отелей.

Связи на контекстной диаграмме:

- Гость → Система: поиск и просмотр отелей.
- Зарегистрированный пользователь → Система: управление бронированиями.
- Менеджер отеля → Система: управление своим отелем.
- Администратор → Система: администрирование.
- Система → Email-сервис: отправка уведомлений.
- Система → Платежная система: проведение платежей.
- Система → Картографический сервис: получение карт и геокодирование.

## 2. Контейнерная диаграмма (C2)

Система декомпозирована на следующие контейнеры (микросервисы, базы данных, брокер сообщений, API Gateway и веб-приложение).

### Веб-приложение (Single-Page Application)
- **Технология:** React
- **Описание:** Обеспечивает пользовательский интерфейс для всех категорий пользователей. Взаимодействует с API Gateway по HTTPS/REST.

### API Gateway
- **Технология:** Spring Cloud Gateway
- **Описание:** Единая точка входа, маршрутизация запросов к микросервисам, аутентификация и авторизация. Направляет запросы к User Service, Hotel Service и Booking Service.

### Микросервисы и их базы данных

#### User Service
- **Технология:** Spring Boot
- **Описание:** Управление пользователями: регистрация, поиск по логину, поиск по маске имени/фамилии.
- **База данных:** User Database (PostgreSQL) – хранит данные пользователей.
- **Взаимодействие:**
  - Читает/пишет в свою БД (JDBC).
  - Публикует события (UserRegistered, UserUpdated) в Message Broker.

#### Hotel Service
- **Технология:** Spring Boot
- **Описание:** Управление отелями: создание, получение списка, поиск по городу, интеграция с картографическим сервисом.
- **База данных:** Hotel Database (PostgreSQL) – хранит данные об отелях.
- **Взаимодействие:**
  - Читает/пишет в свою БД (JDBC).
  - Публикует события (HotelCreated, HotelUpdated) в Message Broker.
  - Вызывает Booking Service для уведомления о новом отеле (HTTPS/REST).
  - Вызывает внешний картографический сервис (HTTPS/REST).

#### Booking Service
- **Технология:** Spring Boot
- **Описание:** Управление бронированиями: создание, отмена, получение бронирований пользователя, проверка доступности.
- **База данных:** Booking Database (PostgreSQL) – хранит данные о бронированиях.
- **Взаимодействие:**
  - Читает/пишет в свою БД (JDBC).
  - Публикует события (BookingCreated, BookingCancelled) в Message Broker.
  - Вызывает Payment Service для проведения платежа (HTTPS/REST).

#### Payment Service
- **Технология:** Spring Boot
- **Описание:** Обработка платежей, интеграция с внешней платёжной системой.
- **База данных:** Payment Database (PostgreSQL) – хранит данные о транзакциях.
- **Взаимодействие:**
  - Читает/пишет в свою БД (JDBC).
  - Вызывает внешнюю платёжную систему (HTTPS/REST).
  - Получает синхронные вызовы от Booking Service.

#### Notification Service
- **Технология:** Spring Boot
- **Описание:** Отправка email-уведомлений пользователям и менеджерам.
- **Взаимодействие:**
  - Подписывается на события из Message Broker (AMQP/Kafka).
  - Вызывает внешний Email-сервис (SMTP/HTTPS).

#### Message Broker
- **Технология:** RabbitMQ / Kafka
- **Описание:** Асинхронная шина событий. Обеспечивает надёжную доставку событий между микросервисами (User, Hotel, Booking → Notification).

## 3. Динамическая диаграмма (сценарий «Создание бронирования»)

Для иллюстрации взаимодействия контейнеров выбран ключевой сценарий: **зарегистрированный пользователь бронирует отель**. Диаграмма показывает последовательность шагов с указанием протоколов и технологий.

### Последовательность шагов

1. **Пользователь** заполняет форму на SPA и нажимает «Забронировать».
2. **SPA** отправляет POST /api/bookings в API Gateway.
3. **API Gateway** маршрутизирует запрос в Booking Service.
4. **Booking Service** проверяет наличие свободных номеров (обращение к своей БД).
5. **Booking Database** возвращает результат.
6. **Booking Service** создаёт запись бронирования со статусом «Зарезервировано» (pending).
7. **Booking Service** отправляет запрос в Payment Service на создание платежа.
8. **Payment Service** создаёт транзакцию со статусом «Ожидает оплаты» в своей БД.
9. **Payment Service** вызывает внешнюю платёжную систему.
10. **Платёжная система** возвращает результат.
11. **Payment Service** обновляет статус транзакции в БД.
12. **Payment Service** возвращает результат в Booking Service.
13. **Booking Service** обновляет статус бронирования на «Забронировано» (или «Отменено» при неудаче).
14. **Booking Service** публикует событие BookingCreated в Message Broker.
15. **Message Broker** доставляет событие в Notification Service.
16. **Notification Service** отправляет email-подтверждение через внешний Email-сервис.
17. **Booking Service** возвращает успешный ответ в API Gateway.
18. **API Gateway** возвращает ответ в SPA.
19. **SPA** отображает пользователю страницу подтверждения.

Все синхронные вызовы выполняются по HTTPS/REST, взаимодействие с базами данных — через JDBC, асинхронные события — через AMQP/Kafka.
