# Анализ производительности


### Hot Paths (часто выполняемые операции)
- `GET /api/v1/hotels` и `GET /api/v1/hotels?city=...` —  просмотр каталога отелей выполняется регулярно и составляет основную читающую нагрузку.
- `GET /api/v1/users/{user_id}/bookings` — пользователи многократно запрашивают свои текущие брони.
- `POST /api/v1/bookings` — создание брони при пиковых нагрузках генерирует всплеск запросов.

### Потенциально медленные операции
- Чтение списка отелей из хранилища (особенно с фильтром по городу).
- Чтение бронирований пользователя.
- Проверка конфликтов дат при создании бронирования (поиск пересечений среди активных броней).


###  Стратегия кеширования: 
Cache-Aside (Lazy Loading) с Redis в качестве хранилища кеша.

### Кешируемые endpoint'ы

| Endpoint | Ключ кеша | TTL | Инвалидация |
|---|---|---|---|
| `GET /api/v1/hotels` (включая фильтр по городу) | `hotels:all` или `hotels:city:{normalized_city}` | 60 секунд | Полный сброс кеша отелей при `POST /api/v1/hotels` |
| `GET /api/v1/users/{user_id}/bookings` | `bookings:{user_id}` | 30 секунд | Сброс кеша конкретного пользователя при `POST /api/v1/bookings` и `DELETE /api/v1/bookings/{booking_id}` |

### Обоснование
- Чтение данных происходит в разы чаще, чем их изменение.
- Небольшой TTL сохраняет баланс между актуальностью информации и разгрузкой хранилища.
- Явная инвалидация после каждой записи исключает возврат устаревших результатов.

## 3. Реализация кеширования

**Защищаемый endpoint:** `POST /api/v1/bookings`

**Алгоритм:** Fixed Window Counter для каждого аутентифицированного пользователя.

### Параметры
- Лимит: 20 запросов за 1 минуту на пользователя.
- Сброс окна: один раз в минуту от первого запроса в текущем окне.

### Поведение API
- При допустимом запросе: стандартная обработка + заголовки лимитов.
- При превышении: HTTP `429 Too Many Requests` + JSON-ошибка + заголовки лимитов.

### Заголовки
- `X-RateLimit-Limit` — максимальное количество запросов в окне.
- `X-RateLimit-Remaining` — оставшееся количество запросов.
- `X-RateLimit-Reset` — Unix timestamp сброса окна.

### Примечание по масштабированию
Счетчик лимитов хранится в Redis, что позволяет разделять лимиты между несколькими экземплярами сервиса.

## 5. Реализация rate limiting

Используется Fixed Window Counter. Ключ в Redis имеет формат `ratelimit:booking:{user_id}:{window_start_unix}`, значение — количество запросов в текущем интервале. TTL ключа совпадает с размером окна (60 секунд).

## 6. Влияние на производительность и мониторинг

### Ожидаемый эффект
- Снижение нагрузки на хранилище за счет кеш-хитов на частых чтениях.
- Снижение пикового давления на запись за счет ограничения частоты `POST /api/v1/bookings`.
- Более предсказуемое поведение сервиса при нагрузочных всплесках.

### Метрики для мониторинга
- Перцентили HTTP-задержек (P50/P95/P99) по endpoint'ам.
- Пропускная способность (RPS).
- Количество и задержка запросов к Redis.
- Метрики кеша: hit rate кеша отелей, hit rate кеша бронирований.
- Метрики rate limiter: количество разрешенных и отклоненных (`429`) запросов.

### Измерение эффективности кеширования
- Hit rate = `hits / (hits + misses)`.
- Сравнение количества обращений к хранилищу до и после включения кеша.
- Рост hit rate должен сопровождаться снижением средней задержки и P95.
