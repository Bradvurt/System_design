# Проектирование документной модели для системы бронирования отелей

## Обоснование выбора между embedded и references

### 1. Коллекция `users`
**Тип: Независимая коллекция (references)**
- **Обоснование**: Пользователи являются независимыми сущностями, которые могут существовать без других документов.
- Доступ к пользователям требуется во многих частях системы (бронирования, аутентификация).

### 2. Коллекция `hotels`
**Тип: Независимая коллекция с embedded `rooms`**
- **Обоснование**:
  - Номера (`rooms`) не имеют смысла без отеля, в котором они находятся.
  - Количество номеров в отеле обычно невелико (десятки-сотни).
  - Доступ к номерам почти всегда происходит в контексте конкретного отеля.
  - Embedded подход позволяет атомарно обновлять информацию об отеле и его номерах.

### 3. Коллекция `bookings`
**Тип: Независимая коллекция с references на `users` и `hotels`**
- **Обоснование**:
  - Бронирования связывают пользователей и отели.
  - Количество бронирований может быть очень большим.
  - Требуется частое обновление статуса бронирования.
  - Необходимость сложных запросов (поиск по датам, пользователям, отелям).
  - References позволяют избежать дублирования данных и проблем с согласованностью.

### 4. Коллекция `sessions`
**Тип: Независимая коллекция**
- **Обоснование**: Сессии имеют собственный жизненный цикл и TTL (время жизни).
